## 1. 产品愿景与定位

### 1.1 产品愿景

让每个人拥有一个安全、私密、会持续成长的 AI 伴侣。用户不需要理解 API Key、YAML 配置、Docker 部署，只需打开 EvoClaw，用自然语言描述需求，AI 助手就能帮你完成任务 -- 并且每一次交互都让它变得更懂你。

### 1.2 核心定位

**"第一个会自我进化的本地 AI 伴侣"**

三个关键词：

| 关键词 | 含义 |
|--------|------|
| **自我进化** | Agent 通过交互记忆、行为反馈、能力自发现持续成长，越用越聪明 |
| **本地** | 数据全部存储在用户设备上，敏感信息永不离开本机，支持离线使用 |
| **伴侣** | 不是冷冰冰的工具，而是有记忆、有个性、能持续陪伴成长的 AI |

### 1.3 市场定位

EvoClaw 定位于 OpenClaw 和 DeerFlow 之间的空白地带：既具备 OpenClaw 的终端用户友好性，又拥有超越两者的安全性和智能进化能力。

| 产品 | 定位 | 核心矛盾 |
|------|------|----------|
| OpenClaw | "Your personal AI" | 功能强大但安全隐患严重（ClawHub 20% 恶意包、明文凭证、3 万+ 公开暴露实例），配置门槛高（15+ 小时初始配置） |
| DeerFlow | "Agent runtime" | 开发者工具，非终端用户产品，无原生应用 |
| **EvoClaw** | **"Self-evolving AI"** | **安全默认、零门槛、越用越聪明** |

### 1.4 为什么是现在

1. **安全危机窗口**: OpenClaw 的安全丑闻（Google 安全工程副总裁公开警告不要安装）为"安全优先"的替代品创造了市场需求
2. **本地模型成熟**: Llama 3、Qwen 2.5 等量化模型已能在消费级硬件（M 系列 Mac、16GB 内存 PC）上流畅运行
3. **用户认知升级**: 越来越多的用户关注 AI 隐私问题，"本地优先"从极客偏好变为主流需求
4. **MCP 协议普及**: Model Context Protocol 生态的快速发展降低了能力扩展的技术门槛

## 3. 核心功能需求

### 3.1 特性一：内置安全机制

> **核心理念**: 安全不是用户需要配置的选项，而是产品出厂自带的基因。

#### 用户故事

> 小王第一次打开 EvoClaw，没有任何安全配置页面。当他导入第一份工作文档时，文档自动以加密形式存储。当一个 Agent 试图访问文件系统时，弹窗提示"该 Agent 想访问你的文件系统，是否允许？" -- 像 iPhone 上的权限弹窗一样简单直观。小王感到安心：不需要当安全专家也能安全使用 AI 助手。

#### 功能拆解

| 子功能 | 描述 | 优先级 |
|--------|------|--------|
| **F1.1 零配置安全默认** | 首次启动即启用加密存储（AES-256）、进程间 mTLS 通信、Agent 沙箱执行环境。用户无需任何手动配置 | **P0** |
| **F1.2 权限分级模型** | 类 iOS 权限系统：Agent 请求能力时弹窗授权。权限分为文件系统、网络访问、系统命令执行、剪贴板、通知五大类。支持"仅本次 / 始终允许 / 始终拒绝"三种授权粒度 | **P0** |
| **F1.3 Skill 签名验证** | 安装 Skill 前自动校验数字签名；签名通过后在沙箱中试运行，检测异常行为（如未声明的网络请求、文件读写越界）；运行期行为审计日志 | **P0** |
| **F1.4 凭证金库** | 集成系统级安全存储：macOS Keychain / Windows Credential Manager / Linux Secret Service。所有 API Key、Token 等敏感信息绝不以明文形式存在于磁盘、内存日志或配置文件中 | **P0** |
| **F1.5 安全仪表盘** | 提供安全状态总览面板：已授权权限清单、Skill 安全评分、最近安全事件日志、一键撤销授权 | **P1** |

#### 验收标准

- [ ] 全新安装后零配置即达到加密存储 + 沙箱执行 + 权限控制的安全基线
- [ ] Agent 首次请求任何敏感权限时，必须弹窗获得用户明确授权
- [ ] 未签名或签名无效的 Skill 无法安装；沙箱试运行发现异常行为时阻止安装并告知用户
- [ ] 通过安全审计工具扫描，不存在明文存储的凭证信息
- [ ] 权限可随时在安全仪表盘中查看和撤销

### 3.2 特性二：语义化 Agent 创建

> **核心理念**: 创建 AI 助手应该像和朋友聊天一样简单，而不是编写配置文件。

#### 用户故事

> 小陈想创建一个"论文阅读助手"。他点击"创建新 Agent"，EvoClaw 问："你想创建什么样的助手？" 小陈说："帮我读论文、提取关键信息、做笔记的助手。"EvoClaw 继续追问："你通常读什么领域的论文？偏好什么样的笔记格式？"几轮对话后，一个定制化的论文助手就创建好了。小陈还可以随时和"半成品"对话测试，不满意就继续调整。

#### 功能拆解

| 子功能 | 描述 | 优先级 |
|--------|------|--------|
| **F2.1 对话式创建引导** | 通过多轮自然语言对话收集用户需求，自动生成 Agent 的 SOUL.md（人格定义：角色、语气、专长、约束）和 MEMORY.md（初始记忆：用户偏好、领域知识） | **P0** |
| **F2.2 模板市场** | 预置常见 Agent 模板：研究助手、编程伙伴、写作助手、生活管家、学习教练等。用户可基于模板一键创建后再通过对话微调 | **P1** |
| **F2.3 实时预览与测试** | 创建过程中随时可和"半成品 Agent"对话测试效果，不满意可回退修改。所见即所得的创建体验 | **P1** |
| **F2.4 Agent 基础模板结构** | 每个 Agent 包含标准化的 SOUL.md（人格定义）和 MEMORY.md（记忆存储），结构清晰、可导出、可分享 | **P0** |
| **F2.5 Agent 导入/导出** | 支持将 Agent（含 SOUL.md + MEMORY.md + 已安装 Skill 列表）打包导出为标准格式，可在其他 EvoClaw 实例上导入 | **P2** |

#### 验收标准

- [ ] 用户通过纯自然语言对话（无需接触任何配置文件）即可完成 Agent 创建
- [ ] 创建引导至少覆盖角色定位、专长领域、语气风格、行为约束四个维度的追问
- [ ] 模板市场提供至少 5 个高质量预置模板
- [ ] 创建过程中可随时切换到测试对话，验证 Agent 行为是否符合预期
- [ ] 生成的 SOUL.md 和 MEMORY.md 结构规范，人类可读

### 3.3 特性三：Agent 自我进化

> **核心理念**: EvoClaw 名字中的 "Evo"（Evolution）就是灵魂 -- Agent 不是静态的工具，而是会持续成长的伙伴。

#### 用户故事

> 老李用 EvoClaw 的编程助手已经两个月了。今天他打开进化日志，看到："本周学会了 3 个新技能：TypeScript 泛型最佳实践、Vitest 单元测试模式、pnpm workspace 配置。代码建议采纳率从 68% 提升到 82%。"老李还注意到，助手已经完全适应了他的代码风格 -- 变量命名偏好 camelCase、注释习惯用中文、喜欢函数式编程范式。这些都是 Agent 从日常交互中自动学到的，老李从未手动配置过任何一条规则。

#### 功能拆解

| 子功能 | 描述 | 优先级 |
|--------|------|--------|
| **F3.1 交互记忆沉淀** | 每次对话结束后，自动提取用户偏好（格式、风格、习惯）、领域知识（术语、概念）、纠正记录（用户修正了什么），结构化写入 MEMORY.md | **P0** |
| **F3.2 行为反馈环** | 用户可对 Agent 的每次回复进行点赞/点踩/文字纠正。负面反馈自动分析原因并调整 Agent 行为权重，正面反馈强化该模式 | **P0** |
| **F3.3 能力图谱** | Agent 自动评估自己在各领域的能力值（如：Python 编程 85/100，英文翻译 72/100）。基于用户交互和反馈动态更新。薄弱领域可主动推荐学习资源或安装增强 Skill | **P1** |
| **F3.4 进化日志** | 用户可查看 Agent 的成长轨迹：新学会的技能、能力值变化、记忆量增长。以时间线形式展示，让用户感受到 Agent 的"成长" | **P1** |
| **F3.5 记忆管理** | 用户可以查看、编辑、删除 Agent 的记忆条目。支持按类别筛选（偏好、知识、纠正）。确保用户对 Agent 的"记忆"有完全控制权 | **P0** |

#### 验收标准

- [ ] 连续 10 次对话后，Agent 的 MEMORY.md 中应包含至少 5 条用户偏好记录
- [ ] 用户对同一类问题连续给出 3 次负面反馈后，Agent 行为应有明显调整
- [ ] 能力图谱至少覆盖 5 个维度，且随交互动态变化
- [ ] 进化日志以周为单位更新，包含可量化的成长指标
- [ ] 用户可在 3 步操作内查看并删除任意记忆条目

### 3.4 特性四：Skill/MCP 自发现机制

> **核心理念**: Agent 不应该因为"缺少某个工具"就无法完成任务，它应该像人一样 -- 发现能力不足时主动去学习。

#### 用户故事

> 小王让 Agent 帮他分析一份 Excel 表格。Agent 发现自己没有处理 Excel 的能力，于是说："我目前还不会处理 Excel 文件。我找到了一个评分 4.8/5 的 'excel-parser' Skill，它可以读取和分析 Excel 数据。需要我安装它吗？"小王点击"安装"，Skill 经过签名验证和沙箱试运行后自动安装完成。30 秒后 Agent 开始分析表格 -- 下次遇到 Excel 就不用再装了。

#### 功能拆解

| 子功能 | 描述 | 优先级 |
|--------|------|--------|
| **F4.1 能力缺口检测** | Agent 执行任务失败或质量不佳时，自动分析失败原因，判断是否为"缺少某种能力"导致 | **P0** |
| **F4.2 多源搜索** | 从多个来源搜索匹配的 Skill/MCP：ClawHub 市场、skills.sh 注册表、npm registry、EvoClaw 自建索引。搜索结果包含评分、下载量、安全评级 | **P1** |
| **F4.3 安全安装流** | 搜索 -> 展示候选列表（含安全评分） -> 用户确认 -> 签名验证 -> 沙箱试运行 -> 正式安装。每一步异常都会中止并告知用户 | **P0** |
| **F4.4 能力编排** | 新安装的 Skill 自动编入 Agent 的能力图谱，Agent 学会在合适的场景调用新能力 | **P1** |
| **F4.5 Skill 推荐引擎** | 基于用户使用模式和 Agent 能力缺口，主动推荐可能有用的 Skill（类似 App Store 推荐），但不强制安装 | **P2** |

#### 验收标准

- [ ] Agent 执行任务失败时，80% 以上的情况能正确识别是否为能力缺口
- [ ] 搜索结果在 3 秒内返回，至少覆盖 2 个以上的 Skill 来源
- [ ] 安全安装流程完整执行，任何一环失败都能正确阻止安装并给出明确原因
- [ ] 新安装的 Skill 在下次同类任务中能被自动调用
- [ ] 用户全程可取消安装流程，Skill 安装不影响 Agent 已有功能

### 3.5 特性五：本地知识库 (RAG)

> **核心理念**: 你的文件就是 AI 的知识 -- 但这些知识永远不会离开你的设备。

#### 用户故事

> 小陈把过去三年的论文笔记（200 多份 Markdown 文件）拖入 EvoClaw 的知识库。EvoClaw 在后台建立向量索引。现在他问"之前读过哪些关于 Transformer 注意力机制优化的论文？"，Agent 精准检索出 8 篇相关笔记，并给出摘要对比。整个过程在本地完成，没有任何数据上传到云端。

#### 功能拆解

| 子功能 | 描述 | 优先级 |
|--------|------|--------|
| **F5.1 文件摄取引擎** | 支持导入本地文件作为知识源：Markdown、PDF、TXT、DOCX、代码文件（.py / .js / .ts 等）。支持拖拽导入和文件夹批量导入 | **P0** |
| **F5.2 本地向量索引** | 使用嵌入式向量数据库（SQLite-vec）在本地建立文件的向量索引。支持增量更新 -- 文件修改后自动重新索引变更部分 | **P0** |
| **F5.3 语义检索** | Agent 对话时自动检索知识库中的相关内容，作为上下文注入 LLM 提示。支持用户手动指定"在 XX 文件夹中搜索" | **P0** |
| **F5.4 知识库管理** | 可视化管理界面：查看已索引文件列表、索引状态、存储占用。支持删除、重新索引、暂停索引 | **P1** |
| **F5.5 多知识库隔离** | 支持创建多个独立知识库（如"工作文档"、"个人笔记"、"代码库"），不同 Agent 可绑定不同知识库 | **P2** |

#### 验收标准

- [ ] 支持至少 5 种文件格式的摄取，单次导入 100 个文件时不超过 5 分钟（M1 Mac 基准）
- [ ] 向量索引全程在本地完成，网络抓包验证无数据外传
- [ ] 语义检索的 Top-5 召回结果与人工标注的相关性匹配率 >= 70%
- [ ] 文件修改后 30 秒内触发增量重新索引
- [ ] 知识库管理界面可在 3 步操作内完成文件的增删管理

### 3.6 特性六：多 Agent 协作网络

> **核心理念**: 一个人再强也不如一个团队 -- Agent 之间的协作让 1+1 > 2。

#### 用户故事

> 老李创建了三个 Agent："研究员"负责搜集技术资料、"架构师"负责系统设计、"码农"负责代码实现。当老李说"帮我调研并实现一个 Redis 缓存方案"时，"研究员"自动搜索最佳实践，将结果推送给"架构师"，"架构师"出设计方案后传递给"码农"编写代码。三个 Agent 协作完成，老李只需在关键节点审核确认。

#### 功能拆解

| 子功能 | 描述 | 优先级 |
|--------|------|--------|
| **F6.1 Agent 间消息通道** | Agent 之间可以发送结构化消息（文本、文件引用、数据对象）。消息传递异步进行，不阻塞用户交互 | **P1** |
| **F6.2 协作工作流定义** | 用户可通过自然语言描述协作流程，系统自动生成 Agent 协作 DAG（有向无环图）。如："研究员找资料 -> 写手写初稿 -> 编辑润色" | **P1** |

| **F6.4 人工审核节点** | 用户可在协作流程中插入"人工审核"节点，关键产出需用户确认后才传递给下一个 Agent | **P1** |
| **F6.5 协作模板** | 预置常见协作模式：研究-写作流水线、需求-设计-开发流水线、翻译-校对流水线 | **P2** |

#### 验收标准

- [ ] 两个 Agent 之间的消息传递延迟 < 1 秒
- [ ] 用户通过自然语言描述可成功创建至少 3 个节点的协作流程
- [ ] 协作流程中单个 Agent 失败不影响其他 Agent 运行，支持失败重试

- [ ] 协作全过程可追溯，用户可查看每个 Agent 的输入输出

### 3.7 特性七：离线优先 + 本地小模型

> **核心理念**: AI 助手不应该因为断网就变成废物。

#### 用户故事

> 小王在飞机上打开 EvoClaw（无网络），让助手帮他整理一份会议纪要。EvoClaw 自动切换到内置的本地小模型（Qwen 2.5 7B 量化版），虽然回答没有 Claude 那么精致，但完全能胜任文本整理工作。落地后连上 WiFi，EvoClaw 无缝切回云端大模型，小王甚至没有察觉切换过程。

#### 功能拆解

| 子功能 | 描述 | 优先级 |
|--------|------|--------|
| **F7.1 本地模型集成** | 内置轻量级本地推理引擎（基于 llama.cpp / MLX），支持运行量化模型（Llama 3 8B / Qwen 2.5 7B 等）。提供一键下载和管理界面 | **P1** |
| **F7.2 云端模型接入** | 支持接入主流云端 LLM：Claude、GPT-4、Gemini、DeepSeek、Qwen 等。用户配置 API Key 后即可使用 | **P0** |
| **F7.3 智能路由** | 根据任务复杂度、网络状态、用户偏好自动选择最优模型。简单任务用本地模型（低延迟、零成本），复杂任务用云端大模型（高质量） | **P1** |
| **F7.4 离线模式** | 断网时自动切换到本地模型，所有核心功能（对话、知识库检索、记忆管理）可离线使用。联网后无缝切回 | **P1** |
| **F7.5 模型管理器** | 可视化的模型管理界面：查看已下载模型列表、模型大小、性能基准。支持下载、删除、切换默认模型 | **P2** |

- [ ] 本地 7B 量化模型在 M1 Mac 上首 Token 延迟 < 500ms，生成速度 >= 15 tokens/s
- [ ] 至少支持 3 家主流云端 LLM Provider 的接入

- [ ] 网络恢复后自动切回云端模型（如用户未修改偏好）
- [ ] 断网状态下对话、知识库检索、记忆读写全部可用

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 要求 | 测试基准 |
|------|------|----------|
| **应用启动时间** | 冷启动 < 3 秒，热启动 < 1 秒 | M1 MacBook Air 8GB |
| **对话响应延迟** | 首 Token 延迟 < 2 秒（云端模型），< 500ms（本地模型） | 网络延迟 < 100ms |
| **知识库索引速度** | 100 个文档（平均 5KB）< 5 分钟 | M1 MacBook Air 8GB |
| **内存占用** | 空闲状态 < 200MB，活跃对话 < 500MB（不含本地模型） | 不含模型权重 |
| **本地模型推理** | >= 15 tokens/s（7B 量化模型） | M1 MacBook Air 16GB |
| **磁盘占用** | 基础安装 < 500MB（不含模型） | 不含向量索引和模型文件 |

### 4.2 安全需求

| 维度 | 要求 |
|------|------|
| **数据加密** | 静态数据 AES-256 加密；传输数据 TLS 1.3 / mTLS |
| **凭证管理** | 全部通过系统 Keychain 存储，禁止明文落盘 |
| **沙箱隔离** | Agent 执行环境沙箱化，文件系统访问需显式授权 |
| **Skill 安全** | 强制签名验证 + 沙箱试运行 + 运行时行为审计 |
| **隐私合规** | 默认无遥测数据收集；用户可选择性开启匿名使用统计 |
| **审计日志** | 记录所有敏感操作（权限变更、Skill 安装、凭证访问），日志本地加密存储 |

### 4.3 可用性需求

| 维度 | 要求 |
|------|------|
| **上手时间** | 新用户首次使用到完成第一次有效对话 < 2 分钟 |
| **零配置原则** | 核心功能（对话、安全、记忆）开箱即用，无需任何手动配置 |
| **错误处理** | 所有错误提供用户可理解的提示和明确的修复建议，禁止暴露技术堆栈信息 |
| **可访问性** | 支持系统级深色/浅色模式；支持键盘导航；字体大小可调 |
| **多语言** | v1.0 支持中文和英文界面 |

### 4.4 兼容性需求

| 平台 | 最低要求 | 目标版本 |
|------|----------|----------|
| **macOS** | macOS 13 (Ventura)+ / Apple Silicon 或 Intel | 原生 Swift 应用 |
| **Windows** | Windows 10 21H2+ / x64 | 原生应用（Electron 或 Tauri） |
| **Linux** | Ubuntu 22.04+ / Fedora 38+ | AppImage / Flatpak |
| **本地模型** | 8GB RAM（7B 量化）/ 16GB RAM（13B 量化） | Apple Silicon 优先优化 |

## 5. 技术约束与假设

### 5.1 技术约束

| 约束 | 说明 |
|------|------|
| **本地优先** | 所有用户数据（对话、记忆、知识库、凭证）必须存储在用户本地设备，除用户主动配置的云端 LLM API 调用外不产生网络数据传输 |
| **跨平台** | 需同时支持 macOS、Windows、Linux 三大桌面平台 |
| **安全底线** | 安全机制不可被用户关闭（可调整授权粒度但不可完全绕过） |
| **离线可用** | 核心功能在无网络环境下必须可用（依赖本地小模型） |
| **模型无关** | 不绑定特定 LLM Provider，支持通过标准接口接入任意兼容模型 |

### 5.2 技术假设

| 假设 | 依据 |
|------|------|
| **本地模型可用性** | 假设 7B 量化模型在消费级硬件（M1 Mac 8GB / RTX 3060）上可提供可接受的推理质量和速度 |
| **SQLite-vec 成熟度** | 假设 SQLite-vec 作为嵌入式向量数据库在万级文档规模下性能和稳定性满足需求 |
| **MCP 协议稳定性** | 假设 Model Context Protocol 在未来 12 个月内保持向后兼容 |
| **系统 Keychain 可用** | 假设目标平台的系统级安全存储 API 均可正常访问 |
| **Skill 生态建设** | 假设可以复用 OpenClaw ClawHub 和 npm 生态的已有 Skill（通过适配层） |

### 5.3 技术选型方向（建议）

| 层次 | 推荐技术 | 备选 |
|------|----------|------|
| **桌面框架** | Tauri 2.0 (Rust + WebView) | Electron |
| **前端** | React + TypeScript + Tailwind CSS | Vue 3 |
| **后端/核心** | Rust (核心引擎) + TypeScript (业务逻辑) | 全 TypeScript |
| **本地推理** | llama.cpp (跨平台) / MLX (Apple Silicon) | Ollama 集成 |
| **向量数据库** | SQLite-vec | LanceDB |
| **本地存储** | SQLite (结构化数据) + 文件系统 (文档/模型) | - |
| **加密** | libsodium / 系统 Keychain API | - |
| **MCP 集成** | MCP SDK (TypeScript) | 自研适配层 |

## 6. MVP 范围定义

### 6.1 MVP 核心原则

> "如果你的第一版不让你感到尴尬，说明你发布得太晚了。" -- Eric Ries

MVP 版本（v0.1）聚焦于一个核心价值验证：**一个开箱即用的、安全的、会记忆的本地 AI 助手**。

### 6.2 MVP 包含（P0 功能）

| 特性 | MVP 范围 |
|------|----------|
| **内置安全机制** | 零配置加密存储 + 权限弹窗授权 + 凭证金库（Keychain 集成） |
| **语义化 Agent 创建** | 对话式创建引导 + SOUL.md / MEMORY.md 自动生成（无模板市场） |
| **Agent 自我进化** | 交互记忆沉淀 + 行为反馈（点赞/点踩） + 记忆管理（查看/删除） |
| **Skill 自发现** | 能力缺口检测 + 安全安装流（单一来源：npm registry） |
| **本地知识库** | 文件摄取（Markdown / TXT / PDF）+ 本地向量索引 + 语义检索 |
| **云端模型接入** | 至少支持 Claude + GPT-4 两家 Provider |

### 6.3 MVP 不包含（明确排除）

| 排除项 | 原因 |
|--------|------|
| **多 Agent 协作网络** | 依赖单 Agent 体验完善后再扩展 |
| **本地小模型 / 离线模式** | 增加安装体积和技术复杂度，v0.5 再加入 |
| **可视化进化仪表盘** | 需积累足够交互数据后才有意义，v0.5 再加入 |
| **模板市场** | 先验证对话式创建流程，市场可后续补充 |
| **Windows / Linux 版本** | 先聚焦 macOS 验证核心体验 |
| **多知识库隔离** | 单知识库足以验证 RAG 价值 |
| **Agent 导入/导出** | 非核心验证项 |
| **Skill 推荐引擎** | 先验证自发现机制，推荐属于增强 |

### 6.4 MVP 成功标准

MVP 成功 = 用户在以下场景中给出正面反馈：

1. **安全感**: "我不需要担心数据安全" -- 零配置即获得安全保障
2. **低门槛**: "5 分钟内我就创建了自己的 AI 助手" -- 纯对话式创建
3. **会记忆**: "它记住了我的偏好，越来越好用" -- 连续使用一周后体验明显改善
4. **能成长**: "它自己找到了缺少的工具并装上了" -- Skill 自发现成功率 > 60%

### 8.2 差异化优势总结

**EvoClaw vs OpenClaw**:
- 安全性从"高风险"提升到"零配置安全默认"
- 上手门槛从 15+ 小时降低到 2 分钟
- 新增自我进化能力（OpenClaw 完全不具备）
- 新增离线使用能力
- 不具备 OpenClaw 的 22+ 消息平台接入能力（有意取舍：聚焦桌面端深度体验）

- 从"开发者工具"转变为"终端用户产品"
- 提供原生桌面应用而非 Web 部署
- 新增语义化 Agent 创建（DeerFlow 需要写 Markdown 配置）
- 新增 Skill 自发现机制

**EvoClaw vs ChatGPT Desktop**:
- 数据全本地，不上传到任何云端服务器
- Agent 可定制化和持续进化（ChatGPT 的记忆功能不透明且不可控）
- 支持离线使用
- 开源可审计
- 不具备 ChatGPT 的模型能力优势（但可通过 API 接入 GPT-4）

### 9.1 Pre-mortem 分析

> 假设现在是 2026 年 12 月，EvoClaw v1.0 发布后完全失败了。最可能的原因是什么？

### 9.2 风险矩阵

| # | 风险 | 类型 | 可能性 | 严重性 | 优先级 | 缓解策略 |
|---|------|------|--------|--------|--------|----------|
| R1 | **本地模型质量不足** -- 用户期望接近 GPT-4 的体验，但本地 7B 模型差距明显，导致失望 | 技术 | 高 | 高 | 关键 | MVP 先以云端模型为主要体验，本地模型定位为"离线备选"而非"主力"；在 UI 上明确标注当前使用的模型及其能力级别；持续跟踪本地模型进步（Llama 4 / Qwen 3 等） |
| R2 | **记忆进化效果不明显** -- 用户使用一周后感觉 Agent "没变聪明"，进化承诺落空 | 产品 | 高 | 高 | 关键 | 设计明确的进化反馈机制（进化日志、能力值变化通知）让用户"感知到"进步；对比展示"第一天 vs 现在"的回答质量差异；设置合理的进化预期（不是一天变天才，而是持续微调适应） |
| R3 | **安全机制影响体验** -- 频繁的权限弹窗让用户烦躁，安全变成了体验障碍 | 产品 | 中 | 高 | 重要 | 设计智能的权限记忆策略："始终允许"减少重复弹窗；权限按场景聚合而非按操作单独申请；参考 macOS 权限弹窗的用户研究最佳实践 |
| R4 | **Skill 生态匮乏** -- 自发现机制很好但搜不到有用的 Skill，巧妇难为无米之炊 | 生态 | 中 | 高 | 重要 | MVP 阶段复用 npm 生态的已有包（通过 MCP 适配层）；优先自建 20-30 个高频 Skill（文件处理、网页抓取、日历、邮件等）；v1.0 后开放 Skill SDK 吸引社区贡献 |
| R5 | **跨平台开发资源不足** -- 同时维护 macOS / Windows / Linux 三个平台拖慢迭代速度 | 资源 | 中 | 中 | 重要 | MVP 只做 macOS；使用 Tauri 2.0 最大化代码复用；Windows / Linux 版本在 v1.0 才加入，且优先保证 macOS 体验 |
| R6 | **隐私承诺难以验证** -- 用户质疑"怎么证明数据没有偷偷上传？" | 信任 | 中 | 中 | 重要 | 全程开源，代码可审计；内置网络活动监控面板，用户可实时查看所有网络请求；邀请第三方安全团队审计并公开报告 |
| R7 | **与 OpenClaw 的竞争** -- OpenClaw 247K stars 的社区优势碾压，EvoClaw 难以获取用户 | 市场 | 中 | 中 | 重要 | 不做全面竞争，聚焦"安全 + 进化"差异点；利用 OpenClaw 安全丑闻作为营销窗口；目标用户是"试过 OpenClaw 但被安全/门槛劝退"的群体 |
| R8 | **知识库 RAG 检索质量差** -- 用户导入大量文件后检索结果不相关，体验糟糕 | 技术 | 中 | 中 | 一般 | 使用高质量的 Embedding 模型；支持用户手动标注检索结果相关性以优化；提供检索范围限定功能（指定文件夹 / 文件类型） |
| R9 | **用户不理解"进化"概念** -- 产品最大卖点但用户不知道怎么触发/利用进化能力 | 产品 | 低 | 高 | 一般 | 首次使用引导（Onboarding）中明确解释进化机制；进化仪表盘作为核心入口持续提醒用户；设计"进化里程碑"成就系统 |
| R10 | **多 Agent 协作复杂度爆炸** -- 多个 Agent 协作时出现消息循环、死锁、结果冲突 | 技术 | 低 | 中 | 一般 | v1.0 协作仅支持 DAG（无环图）工作流；限制单次协作最多 5 个 Agent；设置超时机制防止死锁；人工审核节点作为安全阀 |

## 10. 成功指标 (KPI)

### 10.1 北极星指标

**周活跃用户的 Agent 进化分数中位数**

> 定义: 活跃用户的 Agent 在过去 7 天内的记忆增量 + 能力值变化 + 正面反馈率的综合评分。
> 这个指标同时衡量了用户活跃度和核心价值（进化）的交付效果。

### 10.2 分阶段 KPI

#### v0.1 MVP 阶段 KPI

| 指标 | 目标 | 衡量方式 |
|------|------|----------|
| 种子用户数 | >= 100 | 内测注册数 |
| 首次对话完成率 | >= 80% | 下载后 24 小时内完成首次有效对话的用户比例 |
| 7 日留存率 | >= 40% | 首次使用后第 7 天仍活跃 |
| Agent 创建成功率 | >= 90% | 开始创建流程的用户中成功创建 Agent 的比例 |
| 安全零事故 | 0 起 | 无明文凭证泄露、无未授权数据访问 |
| NPS（净推荐值） | >= 30 | 用户调研 |

#### v0.5 阶段 KPI

| 指标 | 目标 | 衡量方式 |
|------|------|----------|
| 累计用户数 | >= 1,000 | 下载安装数 |
| 30 日留存率 | >= 25% | 首次使用后第 30 天仍活跃 |
| 进化感知率 | >= 60% | 用户调研中回答"Agent 有变聪明"的比例 |
| 离线使用率 | >= 15% | 使用过离线模式的活跃用户比例 |
| 平均 Skill 安装数 | >= 2 | 每个活跃用户安装的 Skill 数量 |
| 知识库使用率 | >= 30% | 创建了知识库的活跃用户比例 |

#### v1.0 阶段 KPI

|------|------|----------|

| 社区贡献者 | >= 30 | GitHub 贡献者数 |

|------|------|----------|

### 10.3 反指标（需要警惕的指标）

| 反指标 | 警戒线 | 说明 |
|--------|--------|------|
| 权限弹窗取消率 | > 40% | 说明弹窗过于频繁或不合理，安全机制在伤害体验 |
| Agent 创建放弃率 | > 30% | 说明对话式创建流程不够流畅 |
| 记忆删除频率 | 日均 > 3 次 / 用户 | 说明记忆沉淀质量差，记了不该记的 |
| Skill 安装失败率 | > 20% | 说明自发现机制推荐了不合适的 Skill 或安装流程有问题 |
| 本地模型弃用率 | > 70% | 说明本地模型质量太差，用户完全不愿意用 |

## 附录 A：术语表

| 术语 | 定义 |
|------|------|
| **Agent** | 具有特定人格、记忆和能力的 AI 助手实例 |

| **SQLite-vec** | SQLite 的向量扩展，支持本地向量检索 |
| **DAG** | Directed Acyclic Graph，有向无环图，描述多 Agent 协作流程 |

| **进化日志** | 记录 Agent 成长轨迹的时间线日志 |
