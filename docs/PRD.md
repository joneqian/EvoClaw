## 1. 产品愿景与定位

### 1.1 产品愿景

让每个人拥有一个安全、私密、会持续成长的 AI 伴侣。用户不需要理解 API Key、YAML 配置、Docker 部署，只需打开 EvoClaw，用自然语言描述需求，AI 助手就能帮你完成任务 -- 并且每一次交互都让它变得更懂你。

### 1.2 核心定位

**"第一个会自我进化的 AI 伴侣"**

三个关键词：

| 关键词 | 含义 |
|--------|------|
| **自我进化** | Agent 通过多层记忆系统（LCM 无损压缩、知识图谱、Hebbian 衰减）、行为反馈、能力自发现持续成长，越用越聪明 |
| **安全** | 数据本地加密存储，凭证系统级保护，权限分级管控，Skill 签名验证，记忆跨通道安全隔离 |
| **伴侣** | 不是冷冰冰的工具，而是有记忆、有个性、能持续陪伴成长的 AI |

### 1.3 市场定位

EvoClaw 定位于 OpenClaw 和 DeerFlow 之间的空白地带：借鉴 OpenClaw 的多层记忆架构思想并做出自有实现，同时拥有超越两者的安全性和零门槛体验。**面向中国国内用户**，深度集成国内主流 IM 平台和大模型服务。

| 产品 | 定位 | 核心矛盾 |
|------|------|----------|
| OpenClaw | "Your personal AI" | 记忆能力强大（12 层架构）但安全隐患严重，配置门槛高（15+ 小时），面向海外用户 |
| DeerFlow | "Agent runtime" | 开发者工具，非终端用户产品，无原生应用 |
| **EvoClaw** | **"Self-evolving AI"** | **借鉴 OpenClaw 记忆思想、安全默认、零门槛、深度适配国内生态** |

### 1.4 为什么是现在

1. **安全危机窗口**: OpenClaw 的安全丑闻（Google 安全工程副总裁公开警告不要安装）为"安全优先"的替代品创造了市场需求
2. **用户认知升级**: 越来越多的用户关注 AI 隐私问题，"安全优先"从极客偏好变为主流需求
3. **MCP 协议普及**: Model Context Protocol 生态的快速发展降低了能力扩展的技术门槛
4. **国内大模型生态成熟**: DeepSeek、Qwen、GLM、MiniMax 等国产大模型质量追平国际水准，API 价格极具竞争力
5. **记忆架构标杆已现**: OpenClaw 的 12 层记忆架构证明了多层记忆系统的可行性和价值，为后来者提供了清晰的技术路线参考

## 3. 核心功能需求

### 3.1 特性一：内置安全机制

> **核心理念**: 安全不是用户需要配置的选项，而是产品出厂自带的基因。

#### 用户故事

> 小王第一次打开 EvoClaw，没有任何安全配置页面。当他导入第一份工作文档时，文档自动以加密形式存储。当一个 Agent 试图访问文件系统时，弹窗提示"该 Agent 想访问你的文件系统，是否允许？" -- 像 iPhone 上的权限弹窗一样简单直观。小王感到安心：不需要当安全专家也能安全使用 AI 助手。

#### 功能拆解

| 子功能 | 描述 | 优先级 |
|--------|------|--------|
| **F1.1 零配置安全默认** | 首次启动即启用加密存储（AES-256）、进程间安全通信、Agent 沙箱执行环境。用户无需任何手动配置 | **P0** |
| **F1.2 权限分级模型** | 类 iOS 权限系统：Agent 请求能力时弹窗授权。权限分为文件系统、网络访问、系统命令执行、剪贴板、通知五大类。支持"仅本次 / 始终允许 / 始终拒绝"三种授权粒度 | **P0** |
| **F1.3 Skill 签名验证** | 安装 Skill 前自动校验数字签名；签名通过后在沙箱中试运行，检测异常行为（如未声明的网络请求、文件读写越界）；运行期行为审计日志 | **P0** |
| **F1.4 凭证金库** | 集成系统级安全存储：macOS Keychain / Windows Credential Manager / Linux Secret Service。所有 API Key、Token 等敏感信息绝不以明文形式存在于磁盘、内存日志或配置文件中 | **P0** |
| **F1.5 安全仪表盘** | 提供安全状态总览面板：已授权权限清单、Skill 安全评分、最近安全事件日志、一键撤销授权 | **P1** |

#### 验收标准

- [ ] 全新安装后零配置即达到加密存储 + 沙箱执行 + 权限控制的安全基线
- [ ] Agent 首次请求任何敏感权限时，必须弹窗获得用户明确授权
- [ ] 未签名或签名无效的 Skill 无法安装；沙箱试运行发现异常行为时阻止安装并告知用户
- [ ] 通过安全审计工具扫描，不存在明文存储的凭证信息
- [ ] 权限可随时在安全仪表盘中查看和撤销

### 3.2 特性二：语义化 Agent 创建

> **核心理念**: 创建 AI 助手应该像和朋友聊天一样简单，而不是编写配置文件。

> 小陈想创建一个"论文阅读助手"。他点击"创建新 Agent"，EvoClaw 问："你想创建什么样的助手？" 小陈说："帮我读论文、提取关键信息、做笔记的助手。"EvoClaw 继续追问："你通常读什么领域的论文？偏好什么样的笔记格式？"几轮对话后，一个定制化的论文助手就创建好了。小陈还可以随时和"半成品"对话测试，不满意就继续调整。

#### 功能拆解

| 子功能 | 描述 | 优先级 |
|--------|------|--------|
| **F2.1 对话式创建引导** | 通过多轮自然语言对话收集用户需求，自动生成 Agent 的 SOUL.md（人格定义：角色、语气、专长、约束）和 MEMORY.md（初始记忆：用户偏好、领域知识） | **P0** |

| **F2.3 实时预览与测试** | 创建过程中随时可和"半成品 Agent"对话测试效果，不满意可回退修改。所见即所得的创建体验 | **P1** |
| **F2.4 Agent 基础模板结构** | 每个 Agent 包含标准化的 SOUL.md（人格定义）和 MEMORY.md（记忆存储），结构清晰、可导出、可分享 | **P0** |
| **F2.5 Agent 导入/导出** | 支持将 Agent（含 SOUL.md + MEMORY.md + 已安装 Skill 列表）打包导出为标准格式，可在其他 EvoClaw 实例上导入 | **P2** |

- [ ] 用户通过纯自然语言对话（无需接触任何配置文件）即可完成 Agent 创建

- [ ] 生成的 SOUL.md 和 MEMORY.md 结构规范，人类可读

### 3.3 特性三：Agent 自我进化 -- 多层记忆系统

> **核心理念**: EvoClaw 名字中的 "Evo"（Evolution）就是灵魂 -- Agent 不是静态的工具，而是会持续成长的伴侣。进化的基础是一套完整的多层记忆系统：无损上下文压缩、结构化知识图谱、Hebbian 衰减机制、跨通道记忆路由，让 Agent 真正具备"越用越聪明"的能力。

#### 用户故事

> 老李用 EvoClaw 的编程助手已经两个月了。今天他打开进化日志，看到："本周学会了 3 个新技能：TypeScript 泛型最佳实践、Vitest 单元测试模式、pnpm workspace 配置。代码建议采纳率从 68% 提升到 82%。"老李还注意到，助手已经完全适应了他的代码风格 -- 变量命名偏好 camelCase、注释习惯用中文、喜欢函数式编程范式。这些都是 Agent 从日常交互中自动学到的，老李从未手动配置过任何一条规则。
>
> 更令他惊喜的是，即使在一段很长的调试对话中，助手也没有"忘掉"开头讨论的架构决策 -- LCM 无损压缩确保了关键上下文不会丢失。而他在飞书群里和同事讨论时，助手的回答从不会暴露他私聊中透露的个人编码偏好 -- 记忆安全隔离确保了隐私边界。

#### 3.3.1 多层记忆架构总览

EvoClaw 的记忆系统借鉴 OpenClaw 12 层记忆架构的设计思想，结合自身技术栈（better-sqlite3 + sqlite-vec + FTS5 单引擎策略）做出适配实现：

| 层级 | 能力名称 | 描述 | 优先级 | 阶段 |
|------|----------|------|--------|------|
| L1 | **消息全量存储** | 所有对话消息全量存入 SQLite messages 表，不丢弃任何原始数据 | **P0** | MVP |
| L2 | **LCM 无损上下文压缩** | 长对话通过摘要 DAG 实现无损压缩，上下文重组时遍历 DAG 重建相关上下文 | **P0** | MVP |
| L3 | **Session Key 路由** | 多通道会话路由策略，同一 Agent 在不同 Channel 的对话按规则共享或隔离 | **P0** | MVP |
| L4 | **记忆安全隔离** | MEMORY.md 仅在私聊中加载，群聊中不暴露个人记忆；不同 Channel 间记忆可见性有明确策略 | **P0** | MVP |
| L5 | **Daily Logs 日期分片** | `memory/YYYY-MM-DD.md` 追加式情景记忆，记录每日关键交互事件 | **P0** | MVP |
| L6 | **交互记忆沉淀** | 对话结束后自动提取用户偏好、领域知识、纠正记录，结构化写入 MEMORY.md | **P0** | MVP |
| L7 | **知识图谱 (facts 表)** | 实体-关系-别名结构，从对话中自动提取结构化知识存入 facts 表 | **P1** | v0.5 |
| L8 | **Hebbian 衰减机制** | Hot/Warm/Cool 三级记忆衰减，memories 表增加 activation/decay_score 列 | **P1** | v0.5 |
| L9 | **混合搜索** | FTS5 全文检索 + sqlite-vec 向量检索，70/30 加权融合，精准召回相关记忆 | **P1** | v0.5 |
| L10 | **Pre-compaction Memory Flush** | 上下文压缩前自动触发静默保存，防止 LCM 压缩过程中关键信息丢失 | **P0** | MVP |
| L11 | **Metabolism 事实提取** | 对话后异步提取结构化事实（实体、关系、属性）到 facts 表 | **P2** | v1.0 |
| L12 | **Growth Vectors / Crystallization** | 成长向量追踪 Agent 能力变化趋势，30+ 天门控结晶化为永久特质写入 SOUL.md | **P2** | v2.0 |

#### 3.3.2 LCM 无损上下文压缩

**问题**: 长对话中 LLM 上下文窗口有限，传统截断方式会丢失关键信息。

**方案**: Lossless Context Management（LCM）-- 消息全量存 SQLite，通过摘要 DAG 实现无损压缩。

**工作流程**:

```
消息流入 → 全量存入 messages 表 → 达到压缩阈值 → 触发 Pre-compaction Flush
  → 调用 ModelRouter 获取用户配置的 LLM → 生成摘要节点 → 写入 summary_dag 表
  → 上下文重组时遍历 DAG 重建相关上下文 → 注入 LLM 提示
```

**关键设计决策**:
- 摘要生成统一走 ModelRouter 调用用户已配置的云端 LLM，不引入本地模型
- 摘要 DAG 节点包含：摘要文本、覆盖的消息 ID 范围、时间戳、关联实体列表
- 上下文重组时按相关性遍历 DAG，而非简单拼接最近 N 条消息
- Pre-compaction Memory Flush 确保压缩前关键信息已持久化到 memories/facts 表

**数据模型**:

```sql
-- 摘要 DAG 表
CREATE TABLE summary_dag (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  parent_id TEXT,                    -- DAG 父节点
  summary_text TEXT NOT NULL,        -- 摘要内容
  msg_range_start INTEGER NOT NULL,  -- 覆盖消息起始 ID
  msg_range_end INTEGER NOT NULL,    -- 覆盖消息结束 ID
  entity_refs TEXT,                  -- JSON: 关联实体列表
  created_at TEXT NOT NULL,
  FOREIGN KEY (conversation_id) REFERENCES conversations(id)
);
```

#### 3.3.3 Session Key 路由

**问题**: 同一 Agent 在桌面端、飞书、企微等多通道的对话需要合理的路由策略（共享 vs 隔离）。

**Session Key 格式**:

```
agent:<agentId>:<channel>:dm:<peerId>      -- 私聊会话
agent:<agentId>:<channel>:group:<groupId>  -- 群聊会话
```

**路由策略**:

| 场景 | 策略 | 说明 |
|------|------|------|
| 同 Agent 同 Channel 私聊 | 共享 | 同一用户在同一平台的私聊共享上下文 |
| 同 Agent 跨 Channel 私聊 | 可配置（默认共享） | 用户可选择桌面端和飞书的对话是否共享记忆 |
| 同 Agent 群聊 | 隔离 | 每个群聊有独立的会话上下文 |
| 不同 Agent | 完全隔离 | Agent 之间的记忆不互通 |

#### 3.3.4 记忆安全隔离

**核心规则**:
- **MEMORY.md 私聊专属**: Agent 的个人记忆（MEMORY.md）仅在私聊中加载和使用，绝不在群聊中暴露
- **群聊仅用公共知识**: 群聊场景中，Agent 仅使用 SOUL.md（人格）+ 该群的历史上下文，不加载用户的个人偏好和私密记忆
- **跨 Channel 可见性策略**: 用户可配置不同 Channel 间的记忆共享级别：
  - **完全共享**: 所有 Channel 的私聊共享全部记忆（默认）
  - **分级共享**: 仅共享通用偏好，不共享领域特定记忆
  - **完全隔离**: 每个 Channel 独立维护记忆

#### 3.3.5 Daily Logs 日期分片

**存储路径**: `~/.evoclaw/agents/{agentId}/memory/YYYY-MM-DD.md`

**设计要点**:
- 追加式写入，每日一个文件，自然形成时间分片
- 记录当日关键交互事件：用户偏好变更、重要决策、新知识点
- 支持按日期范围检索情景记忆
- 旧日志根据 Hebbian 衰减机制逐步降低检索权重

#### 3.3.6 知识图谱 (facts 表)

**数据模型**:

```sql
-- 知识图谱事实表
CREATE TABLE facts (
  id TEXT PRIMARY KEY,
  agent_id TEXT NOT NULL,
  entity TEXT NOT NULL,           -- 实体名称
  relation TEXT NOT NULL,         -- 关系类型 (e.g., "prefers", "works_on", "knows")
  value TEXT NOT NULL,            -- 关系值
  aliases TEXT,                   -- JSON: 实体别名列表
  confidence REAL DEFAULT 1.0,    -- 置信度 (0.0-1.0)
  source_msg_id TEXT,             -- 来源消息 ID
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  activation REAL DEFAULT 1.0,    -- Hebbian 激活度
  decay_score REAL DEFAULT 1.0,   -- 衰减分数
  FOREIGN KEY (agent_id) REFERENCES agents(id)
);

-- FTS5 全文索引
CREATE VIRTUAL TABLE facts_fts USING fts5(entity, relation, value, content=facts);
```

**提取机制**:
- 对话中自动识别结构化事实（"我喜欢用 TypeScript"、"我的项目用的是 pnpm"）
- 通过 ModelRouter 调用 LLM 做实体关系提取
- 新事实与已有事实合并去重，冲突时更新并保留历史版本
- 支持别名体系（"TS" = "TypeScript" = "ts"）

#### 3.3.7 Hebbian 衰减机制

**三级记忆温度**:

| 级别 | 温度 | activation 范围 | 检索权重 | 说明 |
|------|------|-----------------|----------|------|
| **Hot** | 热记忆 | 0.7 - 1.0 | 100% | 近期频繁访问的记忆，优先召回 |
| **Warm** | 温记忆 | 0.3 - 0.7 | 60% | 中期记忆，检索时降权 |
| **Cool** | 冷记忆 | 0.0 - 0.3 | 20% | 长期未访问，仅在精确匹配时召回 |

**衰减规则**:
- 每次记忆被召回并使用，activation += 0.1（上限 1.0）
- 每 24 小时自然衰减：activation *= 0.95
- Cool 记忆 30 天未访问则归档（不删除，仅不参与常规检索）
- 用户可手动"钉住"重要记忆，使其免于衰减

**数据模型扩展**:

```sql
-- memories 表增加衰减字段
ALTER TABLE memories ADD COLUMN activation REAL DEFAULT 1.0;
ALTER TABLE memories ADD COLUMN decay_score REAL DEFAULT 1.0;
ALTER TABLE memories ADD COLUMN last_accessed_at TEXT;
ALTER TABLE memories ADD COLUMN pinned INTEGER DEFAULT 0;
```

#### 3.3.8 混合搜索

**检索策略**: FTS5 全文检索 + sqlite-vec 向量检索，70/30 加权融合。

```
用户查询 → 并行执行:
  ├─ FTS5 全文检索 (权重 0.7) → 精确匹配 + BM25 排序
  └─ sqlite-vec 向量检索 (权重 0.3) → 语义相似度匹配
→ 结果融合 → Hebbian 激活度加权 → 返回 Top-K
```

**设计要点**:
- 全文检索覆盖 memories、facts、daily logs
- 向量检索覆盖 memories 表（向量列由 Embedding 模型生成）
- 融合排序时乘以 Hebbian activation 系数，热记忆优先
- 群聊场景自动过滤私人记忆

#### 3.3.9 Pre-compaction Memory Flush

**触发时机**: 当 LCM 压缩即将执行时，在压缩前自动执行一次静默记忆保存。

**保存内容**:
- 当前对话窗口中尚未持久化的用户偏好
- 未提取的结构化事实
- 重要决策和结论

**设计目的**: 防止 LCM 压缩过程中，尚未来得及沉淀的关键信息仅存在于即将被压缩的原始消息中。Flush 确保这些信息先落盘到 memories/facts 表，再执行压缩。

#### 功能拆解（综合）

| 子功能 | 描述 | 优先级 |
|--------|------|--------|
| **F3.1 消息全量存储 + LCM 压缩** | 消息全量存 SQLite，摘要 DAG 无损压缩，Pre-compaction Flush 防信息丢失。上下文重组时遍历 DAG 重建相关上下文 | **P0** |

## 5. 技术约束与假设

### 5.1 技术约束

| 约束 | 说明 |
|------|------|
| **数据安全** | 所有用户数据（对话、记忆、知识库、凭证）本地加密存储，除 LLM API 调用和 Channel 消息外不产生网络数据传输 |
| **跨平台** | 需同时支持 macOS、Windows、Linux 三大桌面平台 |
| **安全底线** | 安全机制不可被用户关闭（可调整授权粒度但不可完全绕过） |
| **模型无关** | 不绑定特定 LLM Provider，支持通过标准接口接入任意兼容模型 |
| **一体化体验** | 桌面应用内嵌后端服务，用户双击即用，无需命令行操作 |
| **单引擎策略** | better-sqlite3 + sqlite-vec + FTS5 覆盖所有存储需求（结构化数据、向量索引、全文检索），不引入额外数据库引擎 |
| **LLM 统一调用** | 所有需要 LLM 的内部操作（摘要生成、事实提取、知识图谱构建）统一走 ModelRouter，不引入本地模型 |
| **零配置记忆** | 记忆系统（LCM、衰减、Daily Logs）开箱即用，用户无需配置参数 |

### 5.2 技术假设

| 假设 | 依据 |
|------|------|
| **SQLite-vec 成熟度** | 假设 SQLite-vec 作为嵌入式向量数据库在万级文档规模下性能和稳定性满足需求 |
| **FTS5 中文支持** | 假设 FTS5 配合中文分词 tokenizer 能满足中文全文检索需求 |
| **MCP 协议稳定性** | 假设 Model Context Protocol 在未来 12 个月内保持向后兼容 |
| **系统 Keychain 可用** | 假设目标平台的系统级安全存储 API 均可正常访问 |
| **Skill 生态建设** | 假设可以复用 OpenClaw ClawHub 和 npm 生态的已有 Skill（通过适配层） |
| **国内 IM API 稳定** | 假设飞书、企微、QQ 开放平台 API 在未来 12 个月内保持稳定 |
| **LLM 摘要质量** | 假设通过 ModelRouter 调用的云端 LLM 生成的摘要质量足以支撑 LCM 无损压缩 |

### 5.3 技术选型决策

| 层次 | 选型 | 理由 |
|------|------|------|
| **桌面框架** | Tauri 2.0 (Rust + WebView) | 体积小（~15MB vs Electron ~150MB），Rust 安全层 |
| **前端** | React 19 + TypeScript + Tailwind CSS 4 | 生态最大，Tauri 完美支持 |
| **后端/核心** | TypeScript (Node.js >= 22) 作为 Tauri Sidecar | 与前端共享类型，MCP SDK 原生 TypeScript |
| **安全关键路径** | Rust (Tauri Plugin) | 加密、沙箱、签名验证用 Rust 保证内存安全 |
| **模型调用** | Vercel AI SDK (`ai`) | 统一多 Provider 接口，流式响应，Tool Calling 内置 |
| **Agent 框架** | 自研 + DeerFlow 中间件链模式 | 进化引擎是核心差异化，现有框架不提供 |
| **MCP 集成** | @modelcontextprotocol/sdk | 官方 TypeScript SDK |
| **存储引擎** | better-sqlite3 + sqlite-vec + FTS5 | 单引擎覆盖结构化/向量/全文三种检索，零外部依赖 |
| **加密** | libsodium (sodium-native) | 行业标准审计过的加密库 |
| **包管理** | pnpm monorepo + Turborepo | 高效依赖管理，增量构建 |

## 6. MVP 范围定义

### 6.1 MVP 核心原则

> "如果你的第一版不让你感到尴尬，说明你发布得太晚了。" -- Eric Ries

MVP 版本（v0.1）聚焦于一个核心价值验证：**一个开箱即用的、安全的、具备多层记忆的 AI 助手**。

### 6.2 MVP 包含（P0 功能）

| 特性 | MVP 范围 |
|------|----------|
| **内置安全机制** | 零配置加密存储 + 权限弹窗授权 + 凭证金库（Keychain 集成） |

### 6.3 MVP 不包含（明确排除）

|--------|------|

| **Hebbian 衰减 + 混合搜索** | MVP 先用简单检索验证价值，衰减机制 v0.5 加入 |

| **多 Agent 协作网络** | 依赖单 Agent 体验完善后再扩展 |
| **可视化进化仪表盘** | 需积累足够交互数据后才有意义，v0.5 再加入 |

### 6.4 MVP 成功标准

MVP 成功 = 用户在以下场景中给出正面反馈：

1. **安全感**: "我不需要担心数据安全" -- 零配置即获得安全保障
2. **低门槛**: "5 分钟内我就创建了自己的 AI 助手" -- 纯对话式创建
3. **会记忆**: "它记住了我的偏好，越来越好用" -- 连续使用一周后体验明显改善
4. **不忘事**: "长对话也不会忘记之前说过的话" -- LCM 无损压缩验证
5. **能成长**: "它自己找到了缺少的工具并装上了" -- Skill 自发现成功率 > 60%
6. **随处可用**: "在飞书里直接用，不用切应用" -- Channel 接入顺畅
7. **隐私安全**: "群聊里不会暴露我的私人偏好" -- 记忆安全隔离验证

## 9. 风险与缓解策略

### 9.1 Pre-mortem 分析

> 假设现在是 2026 年 12 月，EvoClaw v1.0 发布后完全失败了。最可能的原因是什么？

### 9.2 风险矩阵

| # | 风险 | 类型 | 可能性 | 严重性 | 缓解策略 |
|---|------|------|--------|--------|----------|
| R1 | **记忆进化效果不明显** -- 用户使用一周后感觉 Agent "没变聪明" | 产品 | 中 | 高 | LCM 无损压缩确保长对话不丢上下文；Hebbian 衰减突出高频记忆；知识图谱结构化关键事实；进化日志量化展示成长轨迹；Daily Logs 提供完整情景回溯 |
| R2 | **LCM 摘要质量风险** -- 云端 LLM 生成的摘要丢失关键信息，导致上下文重组不准确 | 技术 | 中 | 高 | Pre-compaction Flush 在压缩前保存关键信息；摘要 DAG 保留原始消息 ID 范围可追溯；支持用户手动触发重新摘要；摘要质量评估指标纳入进化日志 |
| R3 | **记忆衰减参数调优风险** -- Hebbian 衰减过快导致有用记忆被降权，过慢则记忆噪声累积 | 技术 | 中 | 中 | 提供默认参数（衰减系数 0.95/天）但支持用户调整；"钉住"机制让用户保护重要记忆；A/B 测试不同衰减曲线；衰减前检查记忆使用频率 |
| R4 | **安全机制影响体验** -- 频繁的权限弹窗让用户烦躁 | 产品 | 中 | 高 | "始终允许"减少重复弹窗；权限按场景聚合；参考 macOS 权限弹窗最佳实践 |
| R5 | **Skill 生态匮乏** -- 自发现机制很好但搜不到有用的 Skill | 生态 | 中 | 高 | 复用 npm 生态；优先自建 20-30 个高频 Skill；v1.0 后开放 Skill SDK |
| R6 | **跨平台开发资源不足** -- 同时维护三个平台拖慢迭代速度 | 资源 | 中 | 中 | MVP 只做 macOS；Tauri 最大化代码复用 |
| R7 | **隐私承诺难以验证** -- 用户质疑数据安全 | 信任 | 中 | 中 | 全程开源；内置网络活动监控面板；第三方安全审计 |

## 附录 D：记忆层架构概览（OpenClaw 映射表）

EvoClaw 的多层记忆系统借鉴 OpenClaw 12 层架构的设计思想，结合自身技术约束（单引擎 SQLite 策略、云端 LLM 统一调用）做出适配实现：

| # | OpenClaw 原始层 | EvoClaw 适配 | 技术实现 | 差异说明 |
|---|----------------|-------------|----------|----------|
| 1 | **消息全量存储** | L1 消息全量存储 | SQLite messages 表 | 一致 |
| 2 | **LCM 无损压缩** | L2 LCM 无损压缩 | summary_dag 表 + ModelRouter 调用云端 LLM | OpenClaw 用本地模型，EvoClaw 走 ModelRouter 调用用户配置的云端 LLM |
| 3 | **Session Key 路由** | L3 Session Key 路由 | `agent:<id>:<channel>:<scope>:<target>` | 格式一致，EvoClaw 增加跨 Channel 共享策略可配置 |
| 4 | **记忆安全隔离** | L4 记忆安全隔离 | 中间件层 + Session Key 判定 | EvoClaw 增加三级可见性策略（完全共享/分级共享/完全隔离） |
| 5 | **情景记忆** | L5 Daily Logs 日期分片 | `memory/YYYY-MM-DD.md` 文件 | 一致 |
| 6 | **交互记忆沉淀** | L6 交互记忆沉淀 | memories 表 + MEMORY.md 同步 | 一致 |
| 7 | **知识图谱** | L7 知识图谱 (facts 表) | facts 表 + facts_fts (FTS5) | OpenClaw 用独立图数据库，EvoClaw 用 SQLite 单引擎 |
| 8 | **记忆衰减** | L8 Hebbian 衰减 | activation/decay_score 列 | 一致，EvoClaw 增加"钉住"机制 |
| 9 | **混合搜索** | L9 混合搜索 | FTS5 (70%) + sqlite-vec (30%) | OpenClaw 用 Elasticsearch + Pinecone，EvoClaw 用 SQLite 单引擎 |
| 10 | **Pre-compaction Flush** | L10 Pre-compaction Flush | LCM 压缩前触发 | 一致 |
| 11 | **Metabolism** | L11 Metabolism 事实提取 | 异步 Job + ModelRouter | OpenClaw 用专用微服务，EvoClaw 用 Sidecar 内异步任务 |
| 12 | **Growth Vectors + Contemplation** | L12 Growth Vectors / Crystallization | growth_vectors 表 + SOUL.md 写入 | EvoClaw 合并 Contemplation 到 Crystallization 流程，简化架构 |

**设计哲学差异**:
- **OpenClaw**: 分布式微服务架构，每层独立部署，适合大规模云端运行
- **EvoClaw**: 单引擎嵌入式架构，better-sqlite3 + sqlite-vec + FTS5 覆盖全部存储需求，适合桌面端本地运行，零配置开箱即用
