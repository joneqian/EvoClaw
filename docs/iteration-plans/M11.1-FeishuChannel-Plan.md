## 首次真机灰度风险清单（code-reviewer 列出，逐项验证）

mock 测试全绿不代表真机跑通。首次连真实飞书时最可能踩的坑按概率排序：

1. **租户域名**：当前产品只对接国内飞书（`feishu.cn`），client 硬编码 `Lark.Domain.Feishu`。海外 Lark 若未来有需要再恢复（已删除相关前端开关）
2. **WS 长连接心跳 / reconnect 抖动**：SDK 内部 keepalive 超时后自动重连，事件可能短暂重复 dispatch。`normalizeFeishuMessage` 的 `message_id` 可用于幂等，但目前 `handleReceiveMessage` 未做去重 — 建议真机测试时观察是否有重复处理
3. **飞书开发者后台事件白名单**：必须显式勾选 `im.message.receive_v1` / `card.action.trigger` / `drive.notice.comment_add_v1` 等事件，否则永远收不到且**无错误日志**。首次调试最常卡在这里
4. **`receive_id_type` 误判**：`resolveFeishuReceiveId` 去掉 `:sender:`/`:topic:` 后缀依赖 session-key 约定。若 ChannelManager 注入的 peerId 来源不规范（比如第三方工具直接传原始 chat_id），不会被识别为 scope 后缀，但也无损
5. **凭据冷启动链路**：PR1 的 Keychain 加密 → 运行时 decrypt → inject 到 `Lark.Client` 构造，冷启动 + macOS Keychain 首次授权弹窗场景下可能抛错。连接失败时看 `channel_state.credentials` 是否还在原始 JSON 明文

## Followup（未在 4 周范围内，需后续单独立项）

- **凭据加密（systemic）**: 目前所有 Channel（微信 / 企微 / 飞书）的 credentials 都以 JSON 明文存于 `channel_state` 表（SQLite 文件权限 0600）。飞书 `appSecret` / `encryptKey` 是 IM 平台级 bearer，敏感度高于 LLM provider API key（后者已走 M6 Keychain）。建议后续冲刺设计 `ChannelCredentialPool`，让 `channel_state` 只保留句柄，真正凭据走 Keychain + AES-256-GCM（与 M6 Credential Pool 一致）。**归属**: A1 或 M12.1 安全审计冲刺。

## 多机器人群协作增强（对齐 OpenClaw）

### Phase M（已交付，本次 PR1）：Pending History Buffer

移植 OpenClaw `extensions/feishu/src/bot.ts:269-279,482-494,798-813` 的"群旁听缓冲 + 被 @ 时前缀注入"机制。

**场景**：一个群里一个真人 + 多个机器人协作，Bot A 被 @ 时能看到真人和 Bot B 之前说了什么。

**实现**：
- `packages/core/src/channel/adapters/feishu/group-history.ts` — `GroupHistoryBuffer`（内存 rolling + TTL 懒淘汰 + limit FIFO + messageId 去重）
- `inbound.ts`：未 @ 消息进 buffer（原本丢弃）；已 @ 消息 content 前缀注入最近 N 条 + 自身入 buffer
- `index.ts`：`sendMessage` / `sendMediaMessage` 成功后把 Bot 自己的回复回写（默认 on，可关）
- config schema：`groupHistoryEnabled` / `groupHistoryLimit` / `groupHistoryTtlMinutes` / `groupHistoryIncludeBotMessages`（扁平键沿用 `Record<string, string>` 约定）
- `historyKey = chatId`（+ `:topic:threadId`），**不跟随 group_sender** —— buffer 须群级共享才能支持多机器人协作

**默认**：`enabled=true, limit=20, ttlMinutes=30, includeBotMessages=true`，开箱即用。

**测试**：`group-history.test.ts`（16 cases）+ `group-history-integration.test.ts`（10 scenarios）+ `adapter.test.ts` 新增 5 个端到端 case。

### Phase N（已交付，PR2 #待补号）：Broadcast 模式

一条消息 fanout 到多个 Agent，各自独立 session、独立回复。典型场景：AI 圆桌、会议室值守、多专家同时响应。**默认 off**，按需启用。

**架构决策**（与 OpenClaw 原版差异）：
- OpenClaw 靠 `buildBroadcastSessionKey` 把 session key 前缀从 origAgent 换成 targetAgent
- EvoClaw 路由层是"每个 agent 独立 session"，所以直接在 server.ts 入口 fanout，每个 target 用 `generateSessionKey` 构造自己的 key，**不需要字符串替换**
- OpenClaw "active + observer" 模式不复刻：Phase M 的 GroupHistoryBuffer 已经让其他 agent 被 @ 时看得到群前情，无需 observer

**核心模块**：
- `broadcast.ts` — `BroadcastConfig` / `resolveBroadcastTargets` / `extractMentionedAgentIds` / `buildBroadcastDedupeKey`
- 配置：`broadcastEnabled` / `broadcastTriggerMode` ∈ {mention-first, any-mention, always} / `broadcastPeerAgents`（JSON 字符串：`{"chatId": ["agentId-a", ...]}`）
- `ChannelMessage.broadcastTargets?: string[]`（新增字段，由 FeishuAdapter 填，server.ts fanout 时读）
- server.ts 新增 `dispatchBroadcastMessage` helper：for 循环调用 `handleChannelMessage`，失败不影响其他 agent

**激活策略**：
- `mention-first`：只激活被 @ 到的 agent（需要 `botIdToAgentId` 映射，跨 Feishu app 多 bot 时有用）
- `any-mention`（默认）：@ 任一列表 agent 或 @_all 激活全体（最常用，单 Feishu app + 多 agent 共享场景）
- `always`：群内任何被 @ 过滤通过的消息都激活全体

**测试**：`broadcast.test.ts`（单元 26 cases，三档策略 + @_all + 重复去重 + dedupeKey 稳定） + `broadcast-integration.test.ts`（集成 10 cases，SC7-SC12 + 场景组合）

**回滚**：`broadcastEnabled=false` 退化回 BindingRouter 单路路由。
