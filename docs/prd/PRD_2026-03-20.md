## 1. 产品愿景与定位

### 1.1 产品愿景

为企业和团队打造一个安全、可审计、持续进化的 AI Agent 平台。企业 IT 部门无需担心数据泄露或合规风险，开发团队无需花费数周配置环境，团队负责人可以直接在企业 IM（飞书/企微/钉钉）中调用 AI Agent -- 并且每一次交互都让 Agent 变得更懂团队的工作模式和业务上下文。

### 1.2 核心定位

**"企业级自进化 AI Agent 平台"**

四个关键词：

| 关键词 | 含义 |
|--------|------|
| **企业级** | 安全至上、权限精细管控、审计日志完整、数据合规（本地加密存储）、7x24 稳定运行、企业 SIEM 集成 |
| **自进化** | Agent 通过多层记忆系统（L0/L1/L2 三层分级记忆、渐进检索、hotness 衰减、知识图谱）、行为反馈、能力自发现持续成长，越用越聪明 |
| **AI Agent** | 不是简单的 LLM 包装，而是具备记忆、人格、工具使用、任务分解、协作能力的智能体 |
| **平台** | 内置全部能力（Channel、Provider、工具），无需第三方插件，降低企业安全审计和运维成本 |

### 1.3 市场定位

EvoClaw 基于 PI 框架（OpenClaw 底层引擎）构建，在安全性、记忆深度和桌面体验三个维度建立差异化竞争壁垒。**面向中国国内企业和团队用户**，深度集成国内主流企业 IM 平台和大模型服务。

**定位策略：深度取胜，而非广度取胜**

| 维度 | OpenClaw（广度） | EvoClaw（深度） |
|------|-----------------|----------------|
| 能力扩展 | 插件生态（13,700+ Skills，社区驱动） | 内置全部能力，安全审计可控 |
| Channel | 无原生 IM 集成 | 企微/飞书/钉钉/QQ 全部内置 |
| 安全 | 已知安全隐患（Google 安全工程副总裁公开警告） | 零配置安全默认 + 企业合规 |
| 记忆 | 多层记忆（LCM + Growth Vectors） | L0/L1/L2 三层分级 + 渐进检索 + 部门级隔离 |
| 用户 | 开发者/技术爱好者 | 企业 IT 部门、开发团队、团队负责人 |
| 部署 | 需要命令行操作 | 桌面原生，双击即用 |

| 产品 | 定位 | 核心矛盾 |
|------|------|----------|
| OpenClaw | "Your personal AI" | 记忆能力强大但安全隐患严重，配置门槛高（15+ 小时），面向海外个人用户 |
| DeerFlow | "Agent runtime" | 开发者工具，非终端用户产品，无原生应用 |
| 智谱 AutoClaw | "企业级 AI Agent" | 聚焦企业客户但生态封闭，模型绑定自家 |
| 腾讯 WorkBuddy | "工作效率伴侣" | 绑定腾讯生态，跨平台能力受限 |
| 华为小艺Claw | "全场景 AI 助手" | 深度绑定鸿蒙生态，非鸿蒙用户难以使用 |
| 小米 miClaw | "智能生活助手" | 聚焦 IoT 场景，知识工作场景覆盖不足 |
| **EvoClaw** | **"企业级自进化 AI Agent 平台"** | **安全至上 + 深度记忆 + 全内置能力 + 企业 IM 深度集成** |

### 1.4 为什么是现在

1. **安全危机窗口**: OpenClaw 的安全丑闻（Google 安全工程副总裁公开警告不要安装）为"安全优先"的企业级替代品创造了市场需求
2. **企业 AI 采购加速**: 2026 年企业 AI 预算同比增长 80%+，但企业采购部门对安全合规和成本控制提出严格要求
3. **国内企业 IM 生态成熟**: 飞书/企微/钉钉 API 稳定，企业用户习惯已形成，AI Agent 通过 IM 入口可实现零额外学习成本
4. **MCP 协议普及**: Model Context Protocol 生态的快速发展降低了能力扩展的技术门槛
5. **国内大模型生态成熟**: DeepSeek、Qwen、GLM、MiniMax 等国产大模型质量追平国际水准，API 价格极具竞争力
6. **记忆架构研究成熟**: OpenViking L0/L1/L2 三层分级、claude-mem 渐进检索、MemOS 记忆安全协议等研究成果为记忆系统实现提供了清晰的技术路线参考

### 1.5 关键战略决策

| 决策 | 选择 | 理由 |
|------|------|------|
| **无第三方插件系统** | 全部能力内置，无 Plugin SDK | 企业安全审计要求可控的攻击面；插件生态的供应链安全风险对企业不可接受 |
| **无 Web UI** | 桌面原生（macOS → Windows） | 桌面端支持本地加密存储、Keychain 集成、Sidecar 架构；企业 IM Channel 作为移动入口 |
| **自建 Skill 商店** | EvoClaw SkillHub（安全审计后上架） | 企业客户要求 Skill 经过安全审计；ClawHub 生态仍可对接，但 SkillHub 为官方策展源 |
| **Channel 全内置** | 飞书/企微/钉钉/QQ 内置适配器 | 企业 IM 是核心入口，不能依赖第三方插件的稳定性和安全性 |

## 3. 核心功能需求

### 3.1 特性一：企业级安全机制

> **核心理念**: 安全是企业采购的第一道门槛。EvoClaw 以零配置安全默认为基础，叠加企业级安全增强（权限精细管控、Prompt 注入检测、审计日志 SIEM 集成），让企业 IT 部门可以放心部署。

#### 用户故事

> 张总监评估 EvoClaw 时，首先检查安全特性。他发现：所有数据本地 AES-256 加密存储；凭证通过 macOS Keychain 管理；Agent 的每次文件访问、网络请求、命令执行都需要明确授权并留下审计日志；系统内置 17 种 Prompt 注入模式检测，防止恶意指令；审计日志可导出为 Splunk/ELK 兼容格式。张总监确认这些特性后，签字通过了采购审批。

#### 功能拆解

| 子功能 | 描述 | 优先级 |
|--------|------|--------|
| **F1.1 零配置安全默认** | 首次启动即启用加密存储（AES-256-GCM）、进程间安全通信（Bearer Token + 127.0.0.1 绑定）、Agent 沙箱执行环境。用户无需任何手动配置 | **P0** |
| **F1.2 权限分级模型** | 7 类别（文件系统、网络访问、系统命令、剪贴板、通知、Agent 间通信、数据导出）x 4 作用域（once/session/always/deny）。Rust 层集成完成全部权限检查链路，确保无绕过路径 | **P0** |
| **F1.3 Prompt 注入检测** | 内置 17 种注入模式检测：ignore previous instructions、system:、`<\|im_start\|>`、role override、delimiter injection、instruction override、base64 编码逃逸等。对话输入和工具输出双向检测 | **P1** |
| **F1.4 Unicode 混淆检测** | 检测同形字攻击（homoglyph attacks）：拉丁字母与西里尔字母混淆、CJK 统一表意文字混淆、不可见字符注入（零宽空格 U+200B/U+200C/U+200D、方向控制字符 U+200E/U+200F/U+202A-U+202E） | **P1** |
| **F1.5 exec 审批绑定** | 命令执行授权绑定精确 argv 文本，防止"授权 ls 后执行 rm"的提权攻击。用户看到的命令文本 === 实际执行的命令文本 | **P1** |
| **F1.6 沙箱环境变量阻断** | 阻止通过环境变量注入恶意代码：JAVA_TOOL_OPTIONS、PYTHONBREAKPOINT、DOTNET_STARTUP_HOOKS、NODE_OPTIONS、LD_PRELOAD 等。沙箱执行时清除危险环境变量 | **P2** |
| **F1.7 凭证金库** | 集成系统级安全存储：macOS Keychain / Windows Credential Manager。所有 API Key、Token 等敏感信息绝不以明文形式存在于磁盘、内存日志或配置文件中 | **P0** |
| **F1.8 审计日志 SIEM 集成** | 审计日志支持导出为 Splunk HEC 格式和 ELK/Filebeat JSON 格式。记录所有敏感操作（权限变更、Skill 安装、凭证访问、记忆跨通道共享、命令执行）。支持配置日志保留策略 | **P2** |
| **F1.9 数据分类标记** | 对 Agent 处理的数据进行敏感级别标记（公开/内部/机密/绝密），支持基于数据分类的差异化权限策略。满足企业数据合规要求 | **P2** |
| **F1.10 安全仪表盘** | 提供安全状态总览面板：已授权权限清单、最近安全事件日志（含注入检测告警）、一键撤销授权、数据分类统计 | **P1** |
| **F1.11 Skill 安全审计** | 所有 Skill 上架 SkillHub 前经过安全审计。安装时校验数字签名；沙箱试运行检测异常行为（未声明的网络请求、文件读写越界）；运行期行为审计日志 | **P0** |

#### 验收标准

- [ ] 全新安装后零配置即达到加密存储 + 安全通信 + 权限控制的安全基线
- [ ] Agent 首次请求任何敏感权限时，必须弹窗获得用户明确授权
- [ ] 权限模型 7 类别 x 4 作用域全部通过 Rust 层检查，无 Node.js 层绕过路径
- [ ] Prompt 注入检测覆盖 17 种已知模式，检测率 >= 95%
- [ ] Unicode 混淆检测能识别至少 50 种已知同形字对
- [ ] exec 审批：用户授权的命令文本与实际执行文本完全一致
- [ ] 沙箱环境变量阻断覆盖至少 10 种已知危险环境变量
- [ ] 通过安全审计工具扫描，不存在明文存储的凭证信息
- [ ] 审计日志可成功导入 Splunk 和 ELK 系统
- [ ] 权限可随时在安全仪表盘中查看和撤销

### 3.2 特性二：语义化 Agent 创建

> **核心理念**: 创建 AI Agent 应该像和同事交代任务一样简单，而不是编写配置文件。

#### 功能拆解

| 子功能 | 描述 | 优先级 |
|--------|------|--------|
| **F2.1 对话式创建引导** | 通过多轮自然语言对话收集用户需求，自动生成 Agent 工作区的全部 8 个文件：SOUL.md（行为哲学）、IDENTITY.md（外在展示）、AGENTS.md（操作规程）、TOOLS.md（工具文档）、HEARTBEAT.md（周期性行为）、USER.md（用户画像）、MEMORY.md（长期记忆快照）、BOOTSTRAP.md（启动引导） | **P0** |
| **F2.2 企业专家模板** | 预置企业场景 Agent 模板：代码评审助手、会议纪要助手、技术文档助手、项目管理助手、客户支持助手、数据分析助手等。用户可基于模板一键创建后再通过对话微调 | **P1** |
| **F2.3 实时预览与测试** | 创建过程中随时可和"半成品 Agent"对话测试效果，不满意可回退修改。所见即所得的创建体验 | **P1** |
| **F2.4 Agent 工作区模板结构** | 每个 Agent 包含标准化的 8 文件工作区结构（SOUL.md、IDENTITY.md、AGENTS.md、TOOLS.md、HEARTBEAT.md、USER.md、MEMORY.md、BOOTSTRAP.md），可导出、可分享 | **P0** |
| **F2.5 Agent 导入/导出** | 支持将 Agent（含完整工作区文件 + 已安装 Skill 列表）打包导出为标准格式，可在其他 EvoClaw 实例上导入。支持团队内 Agent 共享 | **P2** |
| **F2.6 社区模板兼容** | 兼容 103+ OpenClaw 社区 SOUL.md 模板，用户可直接导入使用或基于社区模板创建 Agent | **P1** |

#### 验收标准

- [ ] 用户通过纯自然语言对话（无需接触任何配置文件）即可完成 Agent 创建
- [ ] 创建引导至少覆盖角色定位、专长领域、语气风格、行为约束四个维度的追问
- [ ] 企业专家模板提供至少 8 个高质量预置模板
- [ ] 创建过程中可随时切换到测试对话，验证 Agent 行为是否符合预期
- [ ] 生成的 8 文件工作区结构规范，人类可读
- [ ] 可成功导入 OpenClaw 社区 SOUL.md 模板并基于其创建 Agent

### 3.3 特性三：Agent 自我进化 -- 多层记忆系统

> **核心理念**: EvoClaw 名字中的 "Evo"（Evolution）就是灵魂 -- Agent 不是静态的工具，而是会持续成长的智能体。进化的基础是一套完整的多层记忆系统：L0/L1/L2 三层分级存储、三阶段渐进检索、hotness 衰减机制、知识图谱、跨通道记忆路由，让 Agent 真正具备"越用越聪明"的能力。

#### 用户故事

> 老李用 EvoClaw 的编程助手已经两个月了。今天他打开进化日志，看到："本周学会了 3 个新技能：TypeScript 泛型最佳实践、Vitest 单元测试模式、pnpm workspace 配置。代码建议采纳率从 68% 提升到 82%。"老李还注意到，助手已经完全适应了团队的代码规范 -- 变量命名偏好 camelCase、注释习惯用中文、喜欢函数式编程范式。这些都是 Agent 从日常交互中自动学到的，团队从未手动配置过任何一条规则。
>

#### 3.3.1 多层记忆架构总览

EvoClaw 的记忆系统借鉴 OpenViking、claude-mem、MemOS 三个项目验证过的核心机制，在 better-sqlite3 单引擎上自主实现：

**核心数据架构 -- 三表协同**:

| 表 | 职责 | 查询模式 |
|----|------|---------|
| **memory_units** | 提炼后的结构化知识，L0/L1/L2 三层存储 | 高频查询，需要快 |
| **knowledge_graph** | 实体间关系网络 | 图查询（subject → predicate → object） |
| **conversation_log** | 原始对话数据，只增不改 | 审计追溯和二次提取 |

**记忆分类体系（9 类）**:

| 分类 | 语义 | 说明 |
|------|------|------|
| profile | merge | 用户基本信息（姓名、职业、所在地） |
| preference | merge | 偏好设定（编程风格、沟通方式） |
| entity | merge | 实体知识：人物/组织/项目 |
| event | independent | 事件/情景记忆 |
| case | independent | Agent 处理过的案例 |
| pattern | merge | 可复用的流程模板 |
| tool | merge | 工具使用经验 |
| skill | merge | 技能/能力沉淀 |
| correction | merge | 用户纠正记录（高优先级） |

**L0/L1/L2 三层分级存储**:

| 层级 | 用途 | Token 量 | 加载时机 |
|------|------|---------|---------|
| **L0** | 一句话摘要，向量检索键 | ~50-100 | Phase 1 宽检索 |
| **L1** | 结构化概览，Markdown 格式 | ~500-2K | Phase 2 精筛 |
| **L2** | 完整内容，含完整背景和细节 | 全文 | Phase 3 按需深加载 |

**merge/independent 语义**: merge 型记忆按 merge_key 去重更新（如用户职业变更时覆盖旧值），independent 型每条独立存储（如每次事件独立记录）。

#### 3.3.2 记忆提取 Pipeline

对话结束后（agent_end 钩子），记忆提取分三个阶段执行：

```
对话结束
    |
    v
+-------------------------------------+
| Stage 1: 预处理（纯逻辑，不调 LLM）    |
| - 剥离注入的记忆上下文（反馈循环防护）  |
|   使用零宽空格标记识别注入内容          |
| - 过滤无信息量的消息（命令、问候等）     |
| - 截断超长工具输出（<=1000 字符）       |
| - CJK 感知的最小长度检查               |
| - [NEW] Rule-based 正则预过滤          |
|   (降低 LLM 调用频次，节省企业 API 成本)|
+---------+---------------------------+
          | 有效内容
          v
+-------------------------------------+
| Stage 2: 记忆提取（一次 LLM 调用）     |
| - 输入：预处理后的对话文本              |
| - 输出：结构化 XML，每条含              |
|   category + merge_key + L0/L1/L2    |
| - 同时输出关系三元组                    |
|   (subject, predicate, object)       |
| - 记忆安全四步裁决：                    |
|   来源验证 -> 归属检查 ->              |
|   置信度评估 -> 隐私边界               |
+---------+---------------------------+
          | ParsedMemory[]
          v
+-------------------------------------+
| Stage 3: 持久化（纯逻辑，不调 LLM）    |
| - merge 型：查 merge_key，存在则更新   |
|   L1/L2，L0 不变（保持向量索引稳定）   |
| - independent 型：直接 INSERT          |
| - 关系三元组写入 knowledge_graph       |
| - 标记已处理的 conversation_log 行     |
| - 异步生成 L0 embedding 写入向量表     |
+-------------------------------------+
```

#### 3.3.3 三阶段渐进检索

每轮对话前（before_agent_start 钩子），通过三阶段渐进加载实现 80%+ token 压缩：

```
用户消息
    |
    v
Phase 1: L0 宽检索（~50ms）
    - FTS5 关键词搜索 l0_index（权重 0.3）
    - sqlite-vec 向量搜索 L0 embedding（权重 0.5）
    - knowledge_graph 关系扩展（权重 0.2）
    - 返回 Top-30 候选 { id, l0, category, activation, score }
    |
    v
Phase 2: 排序 + L1 精筛
    - finalScore = searchScore
      x hotness(activation, access, age)
      x categoryBoost(queryType, category)
      x correctionBoost (correction 类 +0.15)
    - 去重：同 merge_key 只保留最新
    - 可见性过滤（private/shared/channel_only）
    - 取 Top-10，加载 L1 overview
    - 按 category 分组格式化
    |
    v
Phase 3: L2 按需深加载
    - 触发条件（任一满足）：
      a) 用户消息含追问信号（"详细说说/具体是什么/当时怎么..."）
      b) L1 中包含 "[详情已省略]" 标记
      c) category=case 且 queryType=技能型（需要完整案例作为 few-shot）
    - 仅加载触发条件匹配的记忆的 L2
    - Token 预算控制：L2 总量 <= 8K tokens
    |
    v
  组装注入上下文
```

#### 3.3.4 ContextPlugin 生命周期

记忆系统通过 ContextPlugin 接口与 Agent 运行时集成，提供 5 个生命周期钩子：

| 钩子 | 触发时机 | 执行方式 | 用途 |
|------|---------|---------|------|
| **bootstrap** | Agent 首次启动/加载 | 一次性 | 渲染 USER.md / MEMORY.md |
| **beforeTurn** | 每轮对话前 | 串行 | 记忆检索 + 注入上下文 |
| **compact** | token 即将超限时 | 串行（逆序） | 记忆注入从 L1 降级为 L0 |
| **afterTurn** | 每轮对话后 | 并行（异步） | 记忆提取 + 进化评分 |
| **shutdown** | Agent 停止/卸载 | 一次性 | 清理资源 |

**完整插件列表**:

| 插件 | 优先级 | 阶段 | 职责 |
|------|--------|------|------|
| SessionRouterPlugin | 10 | beforeTurn | Session Key 路由 + 可见性范围 |
| PermissionPlugin | 20 | beforeTurn | 权限检查 |
| ContextAssemblerPlugin | 30 | beforeTurn/compact | SOUL.md + USER.md + 历史消息组装 + LCM 压缩 |
| MemoryRecallPlugin | 40 | beforeTurn/compact | 三阶段记忆检索 + 注入 |
| RAGPlugin | 50 | beforeTurn/compact | 知识库语义检索 + 文档注入 |
| ToolRegistryPlugin | 60 | beforeTurn | Tool/Skill/MCP 注册 |
| MemoryExtractPlugin | -- | afterTurn | 记忆提取 pipeline（Stage 1-3） |
| EvolutionPlugin | -- | afterTurn | 进化评分 + 能力图谱更新 |
| GapDetectionPlugin | -- | afterTurn | 能力缺口检测 + Skill 推荐 |
| HeartbeatPlugin | -- | afterTurn | 周期性行为检查 |

#### 3.3.5 Hotness 衰减机制

**衰减公式**:

```
hotness = sigmoid(log1p(access_count)) x exp(-0.099 x age_days)
```

- 半衰期 7 天
- 最低值 0.01（不归零）
- 用户可"钉住"重要记忆，使其免于衰减

**衰减与归档生命周期**:
- 每小时执行一次衰减调度，更新所有非钉选、非归档记忆的 activation 值
- activation < 0.1 且 30 天未访问的记忆归档（不删除，仅不参与常规检索）
- 记忆被召回时自动激活：access_count += 1, activation += 0.1

#### 3.3.6 Session Key 路由

**Session Key 格式**:

```
agent:<agentId>:<channel>:<chatType>:<peerId>
```

**路由策略**:

| 场景 | 策略 | 说明 |
|------|------|------|
| 同 Agent 同 Channel 私聊 | 共享 | 同一用户在同一平台的私聊共享上下文 |
| 同 Agent 跨 Channel 私聊 | 可配置（默认共享） | 用户可选择桌面端和飞书的对话是否共享记忆 |
| 同 Agent 群聊 | 隔离 | 每个群聊有独立的会话上下文 |
| 不同 Agent | 完全隔离 | Agent 之间的记忆不互通（天然支持部门/团队隔离） |

#### 3.3.7 记忆安全隔离

**核心规则**:
- **USER.md / MEMORY.md 私聊专属**: Agent 的个人记忆仅在私聊中加载和使用，绝不在群聊中暴露
- **群聊仅用公共知识**: 群聊场景中，Agent 仅使用 SOUL.md（人格）+ 该群的历史上下文，不加载用户的个人偏好和私密记忆
- **记忆可见性三级控制**: private（仅所属 Agent 私聊可见）、shared（跨 Channel 私聊共享）、channel_only（指定通道可见）
- **部门/团队隔离**: per-Agent 架构天然支持部门级记忆隔离 -- 不同部门使用不同 Agent，记忆完全隔离

**反馈循环防护**:
- 注入的记忆上下文使用零宽空格标记包裹
- 记忆提取时自动剥离标记内的内容，防止注入的记忆被重复存储
- relevance >= 0.45 的记忆才注入上下文

#### 3.3.8 知识图谱

**数据模型**:

```sql
CREATE TABLE knowledge_graph (
  id              TEXT PRIMARY KEY,
  agent_id        TEXT NOT NULL,
  user_id         TEXT NOT NULL,
  subject_id      TEXT NOT NULL,    -- 指向 memory_units.id（entity 类型）
  predicate       TEXT NOT NULL,    -- 关系类型：works_at, knows, uses, prefers...
  object_id       TEXT,             -- 指向另一个 memory_units.id（可选）
  object_literal  TEXT,             -- 或者是字面值
  confidence      REAL NOT NULL DEFAULT 1.0,
  source_memory_id TEXT,
  created_at      INTEGER NOT NULL,
  updated_at      INTEGER NOT NULL

```

**检索集成**: Phase 1 宽检索时，knowledge_graph 参与关系扩展（权重 0.2），如查询提到"EvoClaw"，图查询找到 EvoClaw→uses→TypeScript，把 TypeScript 相关记忆也拉进来。

#### 3.3.9 压缩质量审计（新增）

> **核心理念**: 企业场景要求记忆系统的可靠性可量化，压缩质量审计让管理员能验证 LLM 记忆提取的准确性。

**功能描述**:
- 定期随机抽样对话，对比 LLM 提取前后的内容，量化记忆提取的准确率和召回率
- 生成压缩质量报告：提取遗漏率、分类错误率、L0 摘要质量评分
- 管理员可手动触发全量审计

**优先级**: P2

#### 3.3.10 可配置压缩模型（新增）

> **核心理念**: 企业合规要求数据驻留（data residency），不同企业可能要求记忆提取使用特定 Provider 的模型。

**功能描述**:
- 记忆提取 Pipeline 中的 LLM 调用支持独立配置 Provider 和 Model
- 支持企业指定记忆提取必须走内网部署的模型（如私有化部署的 Qwen/GLM）
- ModelRouter 中增加 `compression_model` 配置项

**优先级**: P2

#### 3.3.11 Rule-based 捕获过滤器（新增）

> **核心理念**: 在 LLM 提取前先用正则规则预过滤，减少无效 LLM 调用，降低企业 API 成本。

**功能描述**:
- 在 Stage 1 预处理中增加可配置的正则规则引擎
- 内置规则：跳过纯代码输出（> 80% 代码行）、跳过 JSON/XML 响应、跳过纯工具调用结果
- 企业管理员可自定义过滤规则
- 统计过滤节省的 API 调用次数和成本

**优先级**: P2

#### 3.3.13 AutoDream 记忆整合

> 记忆只有"生产和消费"还不够，缺少"维护和整合"环节会导致记忆碎片化、矛盾和过期。AutoDream 是后台自动运行的记忆整合器，参考 Claude Code 的 AutoDream 机制。

**触发条件**（从低成本到高成本的渐进门控）:
1. 距上次整合 >= 24 小时
2. 上次整合后 >= 5 个新会话
3. 无其他进程在整合（锁文件检查）

**4 阶段 LLM 驱动整合**:
- **Orient**: 加载所有非归档记忆统计，分析类别分布、重复模式、矛盾信号
- **Gather**: 按类别分组，识别可合并的记忆对/组（同 merge_key 不同版本、语义重叠）
- **Consolidate**: LLM 合并相似记忆、消解矛盾、将相对日期转为绝对日期
- **Prune**: 归档过期/矛盾记忆，清理冗余

**锁机制**: 锁文件 mtime 编码 `lastConsolidatedAt`，多进程安全。锁有效期 1 小时，超时自动接管。

**数据模型**: `consolidation_log` 表记录每次整合的状态、合并/裁剪/创建数量。

**优先级**: P0

#### 3.3.14 会话记忆摘要（Session Memory）

> 独立于上下文压缩的会话级运行摘要，支持长会话回顾和会话恢复。

**触发阈值**:
- 首次：累计 10K tokens
- 后续：每增加 5K tokens 或每 3 次工具调用

**工作方式**:
- 后台 LLM 调用（非阻塞），维护增量式 Markdown 笔记
- 有已有摘要时增量更新，无则全量生成
- 会话恢复时自动注入上次摘要到上下文

**数据模型**: `session_summaries` 表，per-agent per-session_key 唯一，记录摘要文本和触发时的 token/turn/tool_call 计数。

**优先级**: P1

#### 3.3.15 记忆新鲜度保障

> 防止 Agent 过度信任过期记忆。参考 Claude Code 的新鲜度警告机制。

**召回时过期警告**:
- 记忆 > 1 天：追加 `[⚠ N天前]` 标记
- 记忆 > 7 天：追加 `[⚠ 较旧: N天前，建议验证]` 标记

**系统提示词漂移告诫**:
> "记忆可能随时间过期。使用超过 1 天的记忆前，请验证其是否仍然正确。如果记忆与当前状态矛盾，信任当前观察并更新记忆。"

**优先级**: P1

#### 3.3.16 提取互斥与成本优化

> 防止重复提取和降低 API 成本。

**互斥机制**:
- 主 Agent 本轮已操作记忆工具（memory_search/memory_get/knowledge_query） → 提取跳过
- `lastProcessedMsgId` 游标追踪，避免重处理已提取消息
- `inProgress` 标志防止并发提取

**Prompt Cache 共享**:
- 记忆提取 LLM 调用的系统提示词使用 `SystemPromptBlock[]` 格式
- Anthropic 协议：标记 `cache_control: { type: 'ephemeral' }` 复用缓存前缀
- OpenAI 协议：拼接为纯文本（无 cache_control 支持）

**优先级**: P2

#### 3.3.17 LLM 相关性精选

> 在混合搜索 Top-10 结果之后，可选用 LLM 做最终相关性裁决，补齐向量搜索在深度推理场景的不足。

**触发条件**（仅高价值场景）:
- 用户显式召回（"你记得..."、"之前我说过..."、"上次讨论的..."）
- query-analyzer 判定需要详情（needsDetail）

**工作方式**:
- `LlmReranker` 接收 Top-10 候选 + 用户查询
- LLM 返回排序后的 ID 列表
- 替换混合搜索结果

**优先级**: P3

## 5. 技术约束与假设

### 5.1 技术约束

| 约束 | 说明 |
|------|------|
| **数据安全** | 所有用户数据（对话、记忆、知识库、凭证）本地加密存储，除 LLM API 调用和 Channel 消息外不产生网络数据传输 |
| **桌面优先** | 仅支持 macOS 和 Windows 桌面平台，不提供 Web UI 和 Linux 版本 |
| **安全底线** | 安全机制不可被用户关闭（可调整授权粒度但不可完全绕过） |
| **模型无关** | 不绑定特定 LLM Provider，支持通过标准接口接入任意兼容模型 |
| **一体化体验** | 桌面应用内嵌后端服务，用户双击即用，无需命令行操作 |
| **单引擎策略** | better-sqlite3 + sqlite-vec + FTS5 覆盖所有存储需求（结构化数据、向量索引、全文检索），不引入额外数据库引擎 |
| **LLM 统一调用** | 所有需要 LLM 的内部操作（记忆提取、上下文压缩、知识图谱构建）统一走 PI 框架的模型调用，不引入本地模型 |
| **零配置记忆** | 记忆系统（L0/L1/L2 分层、衰减、Daily Logs）开箱即用，用户无需配置参数 |
| **无第三方插件** | 全部能力（Channel、Provider、工具）内置，不开放 Plugin SDK，不支持第三方插件安装。内部 ContextPlugin 5-hook 系统仅用于内部模块化 |
| **Channel 全内置** | 飞书/企微/微信个人号/钉钉/QQ Channel 适配器全部内置实现，不通过插件机制扩展 |

### 5.2 技术假设

| 假设 | 依据 |
|------|------|
| **SQLite-vec 成熟度** | 假设 SQLite-vec 作为嵌入式向量数据库在万级文档规模下性能和稳定性满足需求 |
| **FTS5 中文支持** | 假设 FTS5 配合中文分词 tokenizer 能满足中文全文检索需求 |
| **MCP 协议稳定性** | 假设 Model Context Protocol 在未来 12 个月内保持向后兼容 |
| **系统 Keychain 可用** | 假设目标平台的系统级安全存储 API 均可正常访问 |
| **Skill 生态可用** | EvoClaw SkillHub 自建，初始目录 20-30 个经过安全审计的企业 Skill；ClawHub API 保持可用作为补充源 |
| **国内 IM API 稳定** | 假设飞书、企微、钉钉、QQ 开放平台 API 在未来 12 个月内保持稳定 |
| **LLM 提取质量** | 假设通过 PI 框架调用的云端 LLM 生成的记忆提取和上下文压缩质量满足需求 |
| **企业采购流程** | 假设企业客户接受桌面应用形态的 AI 工具，不强制要求 SaaS 部署 |

### 5.3 技术选型决策

| 层次 | 选型 | 理由 |
|------|------|------|
| **桌面框架** | Tauri 2.0 (Rust + WebView) | 体积小（~15MB vs Electron ~150MB），Rust 安全层 |
| **前端** | React 19 + TypeScript + Tailwind CSS 4 | 生态最大，Tauri 完美支持 |
| **后端/核心** | TypeScript (Node.js >= 22) 作为 Tauri Sidecar | 与前端共享类型，PI 框架原生 TypeScript |
| **安全关键路径** | Rust (Tauri Plugin) | 加密、沙箱、签名验证、权限检查用 Rust 保证内存安全 |
| **模型调用** | PI 框架 (pi-ai + pi-agent-core + pi-coding-agent) | PI 是 OpenClaw 底层引擎，MIT 许可，提供完整 Agent 循环/工具系统/会话持久化/Skills 加载 |
| **Agent 框架** | PI 框架嵌入式运行 + ContextPlugin 扩展 | PI 提供完整的 ReAct 循环、工具执行、会话持久化、auto-compaction；EvoClaw 通过 ContextPlugin 扩展记忆系统和安全层 |
| **MCP 集成** | @modelcontextprotocol/sdk | 官方 TypeScript SDK |
| **存储引擎** | better-sqlite3 + sqlite-vec + FTS5 | 单引擎覆盖结构化/向量/全文三种检索，零外部依赖 |
| **加密** | Rust 层 AES-256-GCM (ring crate) | 安全关键路径在 Rust 层实现，macOS Keychain 通过 security-framework 集成 |
| **沙箱** | Docker（可选，首次触发时引导安装） | 三级安全模式：无沙箱（默认）/ 选择性沙箱 / 全沙箱 |
| **包管理** | pnpm monorepo + Turborepo | 高效依赖管理，增量构建 |

> 假设现在是 2027 年 3 月，EvoClaw v2.0 发布后未能达到企业客户预期。最可能的原因是什么？

### 9.2 风险矩阵

|---|------|------|--------|--------|----------|

## 附录 C：记忆系统架构概览

EvoClaw 的记忆系统借鉴三个 OpenClaw 生态项目验证过的核心机制，在 better-sqlite3 单引擎上自主实现：

| 机制 | 来源 | 核心思想 | EvoClaw 实现 |
|------|------|---------|-------------|
| **L0/L1/L2 三层分级存储** | OpenViking | 每条记忆同时包含索引摘要/结构化概览/完整内容，按需加载 | memory_units 表 l0_index/l1_overview/l2_content 三列 |
| **三阶段渐进检索** | claude-mem | 先搜 L0 定位 → 加载 L1 精筛 → 按需加载 L2 | FTS5 + sqlite-vec 混合搜索 + 分阶段加载 |
| **反馈循环防护 + 相关度阈值** | MemOS | 零宽空格标记防止注入的记忆被重复存储 | 文本清洗 + 标记剥离 |
| **merge/independent 分类策略** | OpenViking | merge 型按 merge_key 去重更新，independent 型独立存储 | category + merge_type + merge_key 字段 |
| **hotness 衰减公式** | OpenViking | sigmoid(log1p(access_count)) x exp(-0.099 x age_days) | activation 列 + 衰减调度器 |
| **记忆安全协议** | MemOS | 四步裁决：来源验证 → 归属检查 → 置信度评估 → 隐私边界 | 提取 Pipeline Stage 2 内置 |

**设计哲学**:
- **单引擎嵌入式架构**: better-sqlite3 + sqlite-vec + FTS5 覆盖全部存储需求，零外部依赖，适合桌面端本地运行
- **PI 框架集成**: 通过 ContextPlugin 扩展钩子与 PI Agent 运行时集成，记忆系统作为 PI 的增强层运行
- **零配置开箱即用**: 记忆系统全自动运行，用户无需理解 L0/L1/L2 分层或配置衰减参数
